# MVP制約システム - 過剰実装防止ガイド

## 🎯 目的

イシュー駆動開発において、Claude Codeが各イシューでMVP範囲を超えた過剰実装を行うことを防ぐシステム。

## 📋 システム概要

### 必須ルール

**🚨 Manager必須行動**：
1. **Workerに指示を出す前に、必ずmvp-principles.mdを読み込む**
2. **タスクがMVP範囲内かを事前検証する**
3. **YAGNI原則違反がないかチェックする**

### ファイル構成

```
docs/mvp-constraints/
├── README.md                 # このファイル
└── mvp-principles.md         # 全イシュー共通のMVP原則
```

## 🛡️ 過剰実装防止原則

### YAGNI（You Aren't Gonna Need It）チェック

タスクを開始する前に以下を自問：

1. **この機能は今すぐ必要か？**
   - 将来的に必要 = 今は実装しない
   - 今必要 = 最小限で実装

2. **この複雑性は本当に必要か？**
   - 複雑 = シンプルに変更
   - シンプル = OK

3. **統計・分析機能を追加していないか？**
   - 統計機能 = 削除
   - 基本機能のみ = OK

## 🚫 禁止パターン

### 絶対に実装してはいけない機能

1. **統計・分析システム**
   - パフォーマンス指標計算
   - 詳細な統計情報
   - 履歴データの長期保存
   - Z-score等の高度な数学的処理

2. **複雑なエラーハンドリング**
   - 詳細なリトライ機構
   - 複雑な状態管理
   - 自動回復システム

3. **過度なプロトコル実装**
   - FIX/RESTの詳細実装
   - 複雑な検証ロジック
   - 高度なフェイルオーバー

4. **未来のための拡張機能**
   - 設定可能なパラメータ
   - プラグインシステム
   - 汎用的なフレームワーク

## ✅ MVP適合チェックリスト

タスク開始前に以下をチェック：

- [ ] 機能は今すぐ必要か？
- [ ] 実装は最小限か？（100行以下が理想）
- [ ] 統計・分析機能を含まないか？
- [ ] 複雑なエラーハンドリングを含まないか？
- [ ] 将来の拡張性を考慮していないか？
- [ ] クラス分離が適切か？（3つ以上に分離していないか）

## 🎯 イシューごとの責任範囲

各イシューは以下のカテゴリに分類され、それぞれのMVP制約が適用されます：

| カテゴリ | 責任範囲 | 禁止事項 |
|---------|---------|---------|
| UI/UX | UI基本機能のみ | 複雑な状態管理、分析機能 |
| Backend | 基本ロジックのみ | パフォーマンス分析、複雑エラー処理 |
| Integration | 基本連携のみ | 分散システム機能、複雑プロトコル |
| Testing | 基本テストのみ | E2E、パフォーマンステスト |

## 🚨 違反時の対応

過剰実装を発見した場合：

1. **即座に実装を停止**
2. **必要最小限に再設計**
3. **mvp-principles.mdを再確認**
4. **CLAUDE.mdの制約を強化**
